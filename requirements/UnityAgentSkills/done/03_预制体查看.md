# 需求文档 - 预制体查看功能

## 1. 项目现状与核心目标

### 1.1 用户需求简述

需要提供两个命令来查询 Unity 预制体的信息：

1. **prefab.queryHierarchy**: 查询预制体内所有 GameObject 的层级结构
2. **prefab.queryComponents**: 查询指定 GameObject 上所有组件的详细信息

### 1.2 项目现状

- 项目已有 UnityAgentSkills 命令系统,支持批量格式,通过 JSON 文件输入和输出.
- 现有 `log.query` 命令可作为实现参考.
- prefab 命令作为插件化扩展提供.

## 2. 范围与边界

### 2.1 功能点简述

本需求包含以下两个独立命令：

#### 命令 1: prefab.queryHierarchy

- [ ] 接收预制体路径(工程相对路径,以 `Assets/` 开头,且为 `Assets/.../*.prefab`),加载并解析预制体.
- [ ] 默认返回预制体内所有 GameObject 的完整层级结构.
- [ ] 每个 GameObject 包含 `name`,`instanceID`,`path`,`siblingIndex`,`depth`,`isActive`,`children`.
- [ ] 支持参数: 是否包含禁用的 GameObject,最大遍历深度.
- [ ] 支持可选参数 `nameContains` 进行名称模糊过滤(包含匹配,忽略大小写,Trim).
- [ ] 当传入 `nameContains` 时,不返回完整 hierarchy 树,改为返回扁平列表 `matches[]`.
- [ ] `matches[]` 每项必须至少包含 `name`,`instanceID`,`path`,`siblingIndex`,`depth`,`isActive`.
- [ ] 现有 `nameContains` 能力继续保留为兼容模式,用于在不知道完整 `objectPath` 时快速缩小范围.
- [ ] 当命中数量很多时,应通过 `maxResults` 控制返回数量,并稳定返回 `totalMatches`,`matchedCount`,`truncated`.
- [ ] 搜索定位体验应对空格,大小写差异友好,并支持后续扩展到更明确的对象搜索入口,但不影响当前 hierarchy 协议兼容性.

#### 命令 2: prefab.queryComponents

- [ ] 接收预制体路径和 GameObject 路径,使用 `objectPath + siblingIndex` 定位目标对象.
- [ ] 返回该 GameObject 上所有组件的详细信息.
- [ ] 每个组件包含: 类型名,instanceID,所有可序列化字段.
- [ ] 支持参数: 组件类型模糊过滤(包含匹配,大小写不敏感),字段详细程度控制.
- [ ] 默认返回所有组件,包括 Transform,RectTransform 等 Unity 内置组件.
- [ ] 查看目标是尽量展示任意组件全部可编辑字段,结果应足以指导后续编辑前分析.
- [ ] 只要字段在 Unity Inspector 中可编辑,就应尽量继续展开其数组/列表元素,嵌套对象字段和多层级子字段.
- [ ] 默认不应因字段复杂,层级深,数量多或内容长而折叠,隐藏,或仅返回空壳容器.
- [ ] 当调用方只关心部分字段时,应优先通过筛选缩小范围,而不是依赖默认折叠.
- [ ] 对暂时无法完全业务语义化的复杂类型,也应返回稳定,一致,可审查的结构,而不是 `UnityEditor.SerializedProperty` 等中间态占位字符串.

### 2.2 排除项

- 不支持直接编辑功能(后续需求).
- 不支持实时运行时对象查询(仅支持预制体文件).
- 不支持基于正则的名称匹配(本需求仅支持 `nameContains` 的 contains 匹配).
- 不要求本阶段支持 AND 搜索,若后续需要必须单独定义参数语义.
- 不要求把所有旧调用方式立即废弃或重命名.
- 不要求为每种组件单独定义关键字段规则,优先采用共享展开规则.
- 不需要预制体文本文件的 fileID(统一使用 Unity InstanceID).
- 不允许以“结果过长”为理由,把本应可见的可编辑字段默认折叠或隐藏.

## 3. 技术实现要点

### 3.1 实现方案

- 使用 Unity 的 SerializedObject 系统操作预制体.
- 返回 InstanceID 用于会话中快速定位.
- 同时记录对象路径作为稳定引用方式.
- 两个命令都遵循项目 UnityAgentSkills 批量命令格式.
- 搜索定位增强应优先服务“先定位对象,再查组件,再编辑”的闭环,返回字段必须可直接复用到后续命令.
- 组件属性展开增强应优先沿用 `SerializedObject/SerializedProperty` 主链路,贴近 Unity Inspector 可编辑结构.
- `propertyFilter` 只影响返回字段范围,不应改变已命中字段的展开深度与结构语义.
- 当结果较长时,主要通过筛选缩小范围,不通过默认折叠,截断或隐藏属性内容.

### 3.2 数据格式

- **输入格式**：遵循 `batchId`, `timeout`, `commands[]` 的批量格式
- **输出格式**：遵循 `results[]`, `status`, `startedAt`, `finishedAt` 的统一响应格式

### 3.3 架构设计要求（为后续扩展准备）

**模块化设计原则**：

- 代码编写时必须采用高度模块化的设计，便于后续扩展到编辑功能和更多资源类型
- 避免将预制体查询的逻辑硬编码在命令处理器中，应抽取为可复用的工具类
- 组件属性读取、层级遍历、对象定位等核心逻辑应独立封装

**可复用性要求**：

- **对象加载与定位模块**：封装通用的预制体加载、实例化、路径定位逻辑
- **SerializedObject 操作模块**：封装统一的属性读取、类型转换、JSON 序列化逻辑
- **层级遍历模块**：封装递归遍历 GameObject 树的工具方法
- **组件查询模块**：封装组件类型过滤、属性过滤的通用逻辑

**后续扩展考虑**：

- 预制体编辑功能（修改组件参数、添加/删除组件等）
- 资产文件查看与编辑（.asset、.shader、.mat、.anim 等）
- 场景文件查询（.unity 场景中的对象层级和组件）
- 资源依赖分析（查询预制体引用的资源文件）

**技术选型说明**：

- 使用 Unity 的 SerializedObject 系统：可以统一访问所有可序列化的对象属性
- 不仅适用于预制体，也适用于 .asset、.shader、.mat 等所有 Unity 可序列化的资源
- 为后续编辑功能提供基础：SerializedObject 同时支持读取和写入属性

#### 使用 nameContains 时的返回形态说明

- 当不传 `nameContains` 时,返回 `hierarchy`(树结构)和 `totalGameObjects`(树中节点总数).
- 当传入 `nameContains` 时,不返回 `hierarchy`,改为返回扁平列表 `matches[]`.
- 此时 `result` 仍保留 `prefabPath` 和 `rootName`,并新增:
  - `totalMatches`: 命中总数(未截断的总命中数)
  - `matchedCount`: 本次实际返回的 matches 条数(<= maxResults)
  - `truncated`: 当 totalMatches > matchedCount 时为 true
- 当匹配数量超过 `maxResults` 时,仅返回前 `maxResults` 条.

`matches[]` 每个元素包含: `name`,`instanceID`,`path`,`depth`,`siblingIndex`,`isActive`.

> 匹配规则: 对入参 `nameContains` 做 Trim,并按 IgnoreCase 进行 contains 匹配.目标 `GameObject.name` 不做 Trim/normalize.

## 5. 参数汇总

### 5.1 prefab.queryHierarchy 参数

| 参数名          | 类型     | 必填 | 默认值 | 说明                                                                                                                                |
| --------------- | -------- | ---- | ------ | ----------------------------------------------------------------------------------------------------------------------------------- |
| prefabPath      | string   | 是   | -      | 预制体工程相对路径(以 `Assets/` 开头,且为 `Assets/.../*.prefab`)                                                                    |
| includeInactive | boolean  | 否   | true   | 是否包含禁用的 GameObject                                                                                                           |
| maxDepth        | int      | 否   | -1     | 最大遍历深度,-1 表示无限                                                                                                            |
| nameContains    | string[] | 否   | null   | 名称模糊过滤(contains),IgnoreCase,对每个词项 Trim,多项 OR.当传入时返回 `matches[]`,不返回 `hierarchy`.                              |
| maxResults      | int      | 否   | 50     | 当传入 `nameContains` 时,最多返回的匹配条数(防止结果过大).超过时截断,并通过 `totalMatches`,`matchedCount`,`truncated` 反映截断情况. |

### 5.2 prefab.queryComponents 参数

| 参数名          | 类型     | 必填 | 默认值 | 说明                                                                |
| --------------- | -------- | ---- | ------ | ------------------------------------------------------------------- |
| prefabPath      | string   | 是   | -      | 预制体工程相对路径(以 `Assets/` 开头,且为 `Assets/.../*.prefab`)    |
| objectPath      | string   | 是   | -      | GameObject 层级路径                                                 |
| siblingIndex    | int      | 否   | 0      | 同名对象序号(同一父节点下同名节点集合内,从 0 开始),用于定位同名对象 |
| componentFilter | string[] | 否   | null   | 组件类型模糊过滤(null 或空数组表示返回所有组件)                     |
| propertyFilter  | string[] | 否   | null   | 组件字段名模糊过滤(null 或空数组表示返回该组件全部字段)             |

**componentFilter 匹配规则**:

- 支持**模糊匹配**(包含匹配,大小写不敏感)
- **OR 关系**: 只要组件名称包含任意一个 filter 字符串就匹配
  - 示例: `["transform", "ui"]` 会匹配 `Transform`, `RectTransform`, `UIButton`, `UILabel` 等
- 对每个词项先做 Trim
- `null`,空数组,或仅包含空字符串时,都视为不传该参数,返回所有组件(包括 `Transform`, `RectTransform` 等内置组件)
- 当传入了有效过滤值但最终没有命中时,正常返回 `components: []`,不报错

**propertyFilter 匹配规则**:

- 支持**模糊匹配**(包含匹配,大小写不敏感)
- **OR 关系**: 只要字段名包含任意一个 filter 字符串就匹配
  - 示例: `["color", "sprite"]` 会匹配 `m_Color`, `m_NormalColor`, `m_Sprite` 等
- 对每个词项先做 Trim
- `null`,空数组,或仅包含空字符串时,都视为不传该参数,返回该组件的全部已采集字段
- 若父字段名命中,则保留该字段对应的整棵嵌套结构
- 若仅子字段命中,返回结构仍应保持稳定,不破坏原有对象层级语义
- `propertyFilter` 只作用于组件内部字段,不参与组件名过滤
- 当 `componentFilter` 与 `propertyFilter` 同时存在时,语义固定为: 先过滤组件,再过滤组件内部字段
- 若组件保留下来,但其字段经过 `propertyFilter` 后没有任何命中,该组件仍可保留,其 `properties` 返回空对象

### 嵌套属性处理问题

**问题描述**:
在`SerializedObjectHelper.GetSerializedProperties`方法中，使用`SerializedProperty.Next(false)`遍历组件属性时，会遍历到嵌套的子属性。例如：

```
m_CorrespondingSourceObject (顶层属性)
  ├── m_CorrespondingSourceObject.m_PrefabInstance (嵌套属性)
  └── m_CorrespondingSourceObject.m_PrefabAsset (嵌套属性)
```

这些嵌套属性都被作为独立的键添加到同一个 JsonData 对象中，导致生成的 JSON 格式错误：

```json
{
  "m_CorrespondingSourceObject" : "m_PrefabInstance" : "m_PrefabAsset" : {
    "instanceID" : 261436,
    "type" : "Object"
  }
}
```

这种格式不是有效的 JSON，包含重复的键名和错误的冒号语法。

**根本原因**:

- `SerializedProperty.Next(false)`会深度遍历所有序列化属性，包括嵌套属性
- 代码中直接将`prop.propertyPath`作为键添加到 JsonData 对象，没有区分顶层属性和嵌套属性
- LitJson2_utf 的 JsonData 对象允许设置重复的键，但生成的 JSON 字符串格式不正确

**可能的影响**:

1. 生成的 JSON 文件无法被正确解析
2. 可能导致 UnityAgentSkills 的 BatchCommandExecutor 无法正确处理返回结果
3. 影响后续的预制体编辑功能，因为无法正确读取组件属性
   **解决方案(选择方案 2)**:

方案 2: 递归构建嵌套 JSON 结构

- 对于嵌套属性，递归构建子对象结构
- 优点: 保留完整属性信息
- 缺点: 实现复杂，需要处理各种特殊情况

**实施步骤**:

1. **设计 propertyPath 解析与 token 模型**

   - 定义 `PathTokenType` 枚举: Field, ArraySize, ArrayIndex (SerializedObjectHelper.cs 12-17 行)
   - 定义 `PathToken` 结构体: Type, FieldName, ArrayIndex (SerializedObjectHelper.cs 22-69 行)
   - 实现三个静态工厂方法: CreateField, CreateArraySize, CreateArrayIndex (SerializedObjectHelper.cs 31-65 行)

2. **实现 ParsePropertyPath 解析逻辑**

   - 解析 Unity 序列化属性路径为 token 列表 (SerializedObjectHelper.cs 206-283 行)
   - 支持格式: "fieldName", "array.size", "array.data[index]", "parent.array.data[0].field"
   - 处理 ".size" 格式转换为 ArraySize token (SerializedObjectHelper.cs 227-240 行)
   - 处理 ".data[index]" 格式转换为 ArrayIndex token (SerializedObjectHelper.cs 243-262 行)

3. **实现 InsertPropertyValue 插入逻辑**

   - 根据 token 列表构建嵌套 JSON 树结构 (SerializedObjectHelper.cs 390-407 行)
   - 支持对象节点创建/复用、数组扩容与占位 null
   - 遍历 token 列表, 在最后一步设置最终值

4. **优化 ConvertSerializedProperty 类型转换**

   - Generic 类型返回空对象容器, 由子属性填充 (SerializedObjectHelper.cs 366-370 行)
   - ManagedReference 返回带类型信息的容器 (SerializedObjectHelper.cs 371-377 行)
   - 避免在 default 分支返回占位符字符串

5. **集成到 GetSerializedProperties 遍历逻辑**
   - 在遍历循环中调用 InsertPropertyValue 替代直接赋值 (SerializedObjectHelper.cs 154-155 行)
   - 实现嵌套属性树结构构建, 保留完整属性信息

**关键代码位置**:

- PathTokenType 枚举定义: SerializedObjectHelper.cs 12-17 行
- PathToken 结构体定义: SerializedObjectHelper.cs 22-69 行
- ParsePropertyPath 方法: SerializedObjectHelper.cs 206-283 行
- InsertPropertyValue 方法: SerializedObjectHelper.cs 390-407 行
- ConvertSerializedProperty (Generic/ManagedReference): SerializedObjectHelper.cs 364-377 行
- GetSerializedProperties 遍历集成: SerializedObjectHelper.cs 100-162 行 (核心调用在 154-155 行)

**方案 2 实现注意事项与局限**:

1. **不使用反射作为主方案**但可使用反射,只要不滥用即可

   - 本实现基于 Unity SerializedObject 系统, 不过度依赖反射 API
   - 优点: 避免滥用反射带来的性能开销, 无需处理私有字段的可见性问题
   - 局限: 只能访问可序列化的字段, 非 SerializeField 的私有字段无法获取
   - 替代方案: 如需访问所有字段, 可补充基于反射的实现作为可选路径
   - **反射使用场景**: 反射仅作为补充路径,用于处理 SerializedObject/SerializedProperty 难以覆盖的少量特殊场景,主属性序列化逻辑仍应优先基于 SerializedObject/SerializedProperty,符合 "不使用反射作为主方案" 的约束

2. **保持 JsonData 类型一致性的重要性**

   - 所有节点必须明确设置 JsonType (Object 或 Array), 否则可能导致插入失败
   - Generic/ManagedReference 等容器类型必须在 ConvertSerializedProperty 中预先创建为 Object 类型
   - 插入数组元素时, 父节点必须确保为 Array 类型 (在 InsertPropertyValue 中处理)
   - 类型不一致会导致 LitJson2_utf 序列化异常或生成错误的 JSON 结构

3. **数组场景避免重复插入**

   - Unity SerializedProperty 遍历时, 同一数组的所有元素会分别遍历
   - 例如: `array.data[0]`, `array.data[1]`, `array.size` 会作为独立 property
   - InsertPropertyValue 通过 ArrayIndex token 区分不同元素, 正确保存在同一数组节点
   - 关键逻辑: 在 SetFinalValue 中检查节点类型, 复用已存在的数组而非重复创建

4. **ManagedReference 输出结构说明**

   - ManagedReference 类型返回稳定模板对象:
     ```json
     {
       "referenceKind": "ManagedReference",
       "isNull": false,
       "managedReferenceFullTypename": "Assembly-CSharp Namespace.TypeName",
       "managedReferenceId": 12345,
       "type": "TypeName",
       "assembly": "Assembly-CSharp"
       // 子属性将继续填充在此对象中
     }
     ```
   - `referenceKind` 用于与 `ObjectReference` 等引用族结果保持统一识别方式
   - `isNull=true` 时表示该托管引用为空, 此时 `managedReferenceFullTypename`,`type`,`assembly` 应为空字符串
   - `managedReferenceFullTypename` 保留 Unity 原始类型标识, 包含程序集信息
   - `managedReferenceId` 为 Unity 内部引用 ID, 用于追踪引用关系
   - `type`,`assembly` 为从 `managedReferenceFullTypename` 派生出的可读性字段
   - 实际子类字段值由后续子属性继续填充, 保持与 Unity Inspector 显示一致

5. **Generic/Array/ManagedReference 类型转换为容器的处理策略**

   - **Generic 类型**:

     - 返回空 Object 容器, 不写任何默认值
     - 由子属性逐步填充, 保持树结构的完整性
     - 避免在 default 分支返回占位符字符串导致结构破坏

   - **Array 类型**:

     - ConvertSerializedProperty 返回 ArraySize token 对应的整数 (数组长度)
     - 数组元素由后续 `array.data[index]` 子属性填充
     - InsertPropertyValue 自动将父节点转换为 Array 类型并扩容

   - **ManagedReference 类型**:

   - 返回稳定模板 Object 容器, 基础字段至少包含 `referenceKind`,`isNull`,`managedReferenceFullTypename`,`managedReferenceId`,`type`,`assembly`
   - 其中 `managedReferenceFullTypename` 与 `managedReferenceId` 由 ConvertSerializedProperty 一次性设置, `type`,`assembly` 作为可读性派生字段补充
   - 实际对象字段由后续子属性继续填充, 保持类型信息与具体子类字段的分离与共存

   - **统一策略**:
     - 所有容器类型都不在 ConvertSerializedProperty 中写默认字符串
     - 避免在 InsertPropertyValue 中触发类型冲突
     - 确保树结构按深度优先顺序正确构建

发现问题:

当预制体中存在同路径同名称的多个 GameObject 时(例如 Panel_Content 下有多个 UIButton_Confirm)

## 同名 GameObject 定位问题

### 问题描述

当预制体中存在**同一路径下同名**的多个 GameObject 时,当前的 `objectPath` 定位方式存在问题。

**示例场景**:

```
DialogMain/
  └─ Panel_Content/
      ├─ UIButton_Confirm  (第一个, siblingIndex=0)
      ├─ UIButton_Confirm  (第二个, siblingIndex=1)
      └─ UIButton_Cancel
```

**当前实现的问题**:

- 当前 `GameObjectPathFinder.FindByPath` 使用 `name == parts[i]` 匹配
- 遍历子对象时 `break` 只返回第一个匹配对象
- 无法精确定位第二个或后续的同名对象

### 解决方案: siblingIndex 参数

#### 设计原则

1. **单一职责**: 不引入 instanceID,避免两套方案增加复杂度
2. **参数分离**: 使用独立的 `siblingIndex` 参数,而非路径语法 `objectPath[1]`
   - 避免与实际 GameObject 名称混淆
   - 参数结构更清晰,便于 Agent 理解
3. **向后兼容**: `siblingIndex` 为可选参数,默认值为 0

#### 方案详情

##### 1. prefab.queryHierarchy 返回增强

**原有返回**:

```json
{
  "name": "UIButton_Confirm",
  "instanceID": 345678901,
  "path": "DialogMain/Panel_Content/UIButton_Confirm",
  "depth": 2,
  "isActive": true
}
```

**增强后返回** (新增 `siblingIndex` 字段):

```json
{
  "name": "UIButton_Confirm",
  "instanceID": 345678901,
  "path": "DialogMain/Panel_Content/UIButton_Confirm",
  "siblingIndex": 0,
  "depth": 2,
  "isActive": true
}
```

**siblingIndex 说明**:

- 定义: 在同一父节点下,按子物体遍历顺序统计的"同名对象序号".
- 范围: 从 0 开始 (第一个同名对象为 0,第二个同名对象为 1,以此类推)
- 跨会话稳定: 序号基于当前层级结构,不随 Unity Editor 重启变化
- 用途: 精确定位同路径下的同名对象

##### 2. prefab.queryComponents 参数增强

**原有参数**:

```json
{
  "prefabPath": "Assets/.../DialogMain.prefab",
  "objectPath": "DialogMain/Panel_Content/UIButton_Confirm"
}
```

**增强后参数** (新增可选的 `siblingIndex`):

```json
{
  "prefabPath": "Assets/.../DialogMain.prefab",
  "objectPath": "DialogMain/Panel_Content/UIButton_Confirm",
  "siblingIndex": 1 // 可选,默认为 0
}
```

**参数行为**:

- **不传 `siblingIndex`**: 默认为 0,定位第一个匹配的对象 (向后兼容)
- **传入 `siblingIndex: 0`**: 显式定位第一个匹配对象
- **传入 `siblingIndex: 1`**: 定位第二个匹配对象

##### 3. prefab.setGameObjectProperties 参数增强 (后续需求)

**参数格式**:

```json
{
  "type": "prefab.setGameObjectProperties",
  "params": {
    "prefabPath": "Assets/.../DialogMain.prefab",
    "objectPath": "DialogMain/Panel_Content/UIButton_Confirm",
    "siblingIndex": 0, // 可选,默认为 0
    "properties": {
      "name": "ConfirmButton_Updated"
    }
  }
}
```

### 对象定位与 siblingIndex 规则说明

当通过 `objectPath + siblingIndex` 定位对象时,必须满足:

- `objectPath` 的每一段按名称匹配.
- 仅在 `objectPath` 的最后一段,才使用 `siblingIndex` 来区分同名对象.
  - 例: `Panel/Btn` 下有 3 个同名 `Btn`,当 `siblingIndex = 1` 时,表示命中第 2 个 `Btn`.
- 当未传 `siblingIndex` 时,默认 `siblingIndex = 0`.

同时,在输出 hierarchy 或 matches 时:

- 必须为每个节点提供 `siblingIndex` 字段.
- `siblingIndex` 的含义是: 在同一个父节点下,同名兄弟节点的序号(从 0 开始计数).

### 示例场景

#### 场景 1: 查询同名对象

**输入**:

```json
{
  "type": "prefab.queryComponents",
  "params": {
    "prefabPath": "Assets/.../DialogMain.prefab",
    "objectPath": "DialogMain/Panel_Content/UIButton_Confirm",
    "siblingIndex": 1 // 查询第二个 UIButton_Confirm
  }
}
```

**层级结构**:

```
Panel_Content/
  ├─ UIButton_Confirm  (siblingIndex=0) ← 跳过
  ├─ UILabel_Title     (不同名,不参与 UIButton_Confirm 的 siblingIndex 计数)
  ├─ UIButton_Confirm  (siblingIndex=1) ← 匹配目标
  └─ UIButton_Cancel   (不同名,不参与 UIButton_Confirm 的 siblingIndex 计数)
```

**结果**: 返回第二个 `UIButton_Confirm` 的组件信息

#### 场景 2: 向后兼容 (不传 siblingIndex)

**输入**:

```json
{
  "objectPath": "DialogMain/Panel_Content/UIButton_Confirm"
  // 不传 siblingIndex
}
```

**行为**: 默认 `siblingIndex = 0`,返回第一个匹配对象 (与现有行为一致)

#### 场景 3: Agent 工作流

1. **查询层级**:

   ```bash
   prefab.queryHierarchy("DialogMain.prefab")
   ```

2. **获取结果**:

   ```json
   {
     "name": "UIButton_Confirm",
     "path": "DialogMain/Panel_Content/UIButton_Confirm",
     "siblingIndex": 1 // Agent 记录此值
   }
   ```

3. **后续操作** (查询组件或编辑):
   ```bash
   prefab.queryComponents(
     "DialogMain.prefab",
     "DialogMain/Panel_Content/UIButton_Confirm",
     siblingIndex=1  // 使用步骤2中记录的值
   )
   ```
