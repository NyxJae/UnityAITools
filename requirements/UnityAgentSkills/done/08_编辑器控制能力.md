# 需求文档 - 编辑器控制能力

## 1. 项目现状与核心目标

### 1.1 用户需求简述

用户希望在现有 `editor.runAction` 之外,新增一组更适合 AI 稳定使用的编辑器控制命令,用于读取 Unity Editor 当前状态,建立编辑器上下文,执行常见编辑器级操作,并在必要时进行回退.

本次新需求与 `07_运行编辑器命令.md` 的关系如下:

- `07_运行编辑器命令.md` 继续负责 `editor.runAction` 这一类"运行已有编辑器动作"的能力.
- 本文档负责新增一组显式 `editor.*` 命令,用于让 AI 更稳定地感知,控制,回退 Unity Editor 状态.
- 本文档不重复定义 `playmode.start`,`playmode.stop`,`playmode.waitFor` 等已有 Play Mode 技能命令,仅在需要时引用其关系.

本次需求的核心目标不是再增加一个"万能调用器",而是提供一组对非专业人士也容易理解,对 AI 也容易稳定调用的编辑器控制面.

### 1.2 当前项目现状

当前项目已经具备:

- `editor.runAction`: 运行编辑器动作,适合触发已有 `public static` 方法.
- `log.query`,`log.screenshot`,`log.refresh`: 负责日志,截图,刷新等待.
- `scene.*`,`prefab.*`,`playmode.*`: 已覆盖内容查看,编辑,Play Mode UI 自动化等能力.

当前不足在于:

- AI 还缺少统一的编辑器状态读取入口.
- AI 还不能稳定读取当前选中对象和编辑器上下文.
- AI 还缺少编辑器级 `undo/redo` 这样的全局回退能力.
- AI 还缺少显式的菜单执行,标签列表,层列表,暂停出错等编辑器控制能力.

### 1.3 本文档覆盖的新增能力

本文档新增并冻结以下命令方向:

1. `editor.getState`
2. `editor.getContext`
3. `editor.select`
4. `editor.getSelection`
5. `editor.undo`
6. `editor.redo`
7. `editor.executeMenu`
8. `editor.getTags`
9. `editor.getLayers`
10. `editor.setPauseOnError`

这些命令与 `editor.runAction` 属于同一大组"编辑器控制能力",但职责不同:

- `editor.runAction` 负责调用已有编辑器动作.
- 上述新命令负责提供稳定,显式,统一语义的编辑器控制面.

## 2. 范围与边界

### 2.1 功能范围

**功能点简述**:

- [ ] 提供统一的编辑器状态读取能力.
- [ ] 提供统一的编辑器上下文读取能力.
- [ ] 提供稳定的选中对象设置与读取能力.
- [ ] 提供编辑器级撤销与重做能力.
- [ ] 提供安全的菜单执行能力.
- [ ] 提供项目全局 tags 和 layers 读取能力.
- [ ] 提供 Pause On Error 开关控制能力.
- [ ] 所有新命令继续沿用现有 pending/results 批次协议.
- [ ] 所有新命令的结果结构必须对非专业人士可读.

### 2.2 与现有 07 文档的边界

本文档必须长期遵守以下边界:

- `07_运行编辑器命令.md` 不删除,继续作为 `editor.runAction` 的需求真源.
- 本文档不把 `editor.runAction` 改名,也不替换其现有职责.
- 本文档新增的是显式 `editor.*` 控制命令,而不是扩展 `actionId` 语义.
- 当某个能力已经可以通过显式 `editor.*` 命令稳定表达时,不应要求调用方再绕到 `editor.runAction`.
- 当某个能力本质上是项目自定义工具动作,并不具备稳定通用协议时,仍应优先留在 `editor.runAction`.

#### 07 与 08 的使用判定

为避免调用方在 `editor.executeMenu` 与 `editor.runAction` 之间反复猜测,本组文档明确采用以下判定:

- 已知目标是某个稳定菜单路径,并且调用方要执行的是"菜单项本身",优先使用 `editor.executeMenu`.
- 已知目标是项目中已暴露的 `actionId`,或目标并不是稳定菜单项,而是某个项目私有工具方法,优先使用 `editor.runAction`.
- 不应为了执行一个已有稳定菜单路径的能力,再额外要求调用方先查方法全名再走 `editor.runAction`.
- `editor.executeMenu` 面向"我知道要点哪个菜单"的场景.
- `editor.runAction` 面向"我知道要执行哪个已暴露动作"的场景.

### 2.3 排除项

**排除项**:

- 不在本文档中重新定义 `playmode.start`,`playmode.stop`,`playmode.queryUI` 等已有 Play Mode 命令.
- 不在本文档中定义 Scene 查看与编辑协议,这些仍由 `05_场景查看.md` 与 `06_场景编辑.md` 负责.
- 不在本文档中定义 Prefab 查看与编辑协议,这些仍由 `03_预制体查看.md` 与 `04_预制体编辑*.md` 负责.
- 不把 `editor.runAction` 做成新命令的兼容壳.
- 不承诺实现"任意反射调用 Unity 任意内部 API".
- 不在本文档中定义工作流级持久化回滚,例如跨会话任务回滚历史.本文档仅覆盖编辑器级即时 `undo/redo`.

## 3. 统一协议要求

### 3.1 顶层输入结构

所有新命令沿用现有批次协议:

- `batchId`: 批次唯一标识.
- `timeout`: 可选,沿用框架统一超时口径.
- `commands`: 命令数组.

每条命令沿用统一结构:

- `id`: 命令唯一标识.
- `type`: 对应具体 `editor.*` 命令名.
- `params`: 参数对象.

### 3.2 统一输出要求

- 批次级状态继续沿用 `processing/completed/error`.
- 命令级状态继续沿用 `success/error`.
- 成功结果必须优先返回可读字段,不能只返回模糊文案.
- 失败时必须提供稳定 `error.code`.
- `error.message` 应优先面向非专业人士可读,避免纯内部术语.

### 3.3 错误码要求

本组新命令至少应复用或覆盖以下错误语义:

- `INVALID_FIELDS`: 参数缺失,参数类型错误,字段格式非法.
- `RUNTIME_ERROR`: Unity 编辑器内部执行失败,或底层 API 返回异常.
- `ONLY_ALLOWED_IN_EDIT_MODE`: 某些命令若仅允许 Edit Mode,则必须使用该错误码.
- `GAME_OBJECT_NOT_FOUND`: 需要按对象定位时,目标对象不存在.
- `ASSET_NOT_FOUND`: 需要按资源路径定位时,目标资源不存在.
- `AMBIGUOUS_TARGET`: 定位信息不足,或目标存在歧义,系统不能安全猜测时使用.
- `MENU_ITEM_NOT_FOUND`: 菜单路径不存在.
- `MENU_ITEM_NOT_EXECUTABLE`: 菜单存在,但当前上下文下不可执行.
- `FORBIDDEN_EDITOR_ACTION`: 命中高风险菜单或受限编辑器操作,被安全拦截.

错误语义说明:

- `GAME_OBJECT_NOT_FOUND` 主要用于场景对象选择失败.
- `ASSET_NOT_FOUND` 主要用于 Project 资源选择失败.
- `AMBIGUOUS_TARGET` 用于同名对象无法稳定定位,或调用方给出的信息不足以安全确定唯一目标.
- `MENU_ITEM_NOT_FOUND` 与 `MENU_ITEM_NOT_EXECUTABLE` 必须区分,避免调用方无法判断是菜单路径写错,还是当前上下文不允许执行.
- `FORBIDDEN_EDITOR_ACTION` 用于高风险菜单或明确受限的编辑器动作拦截,不应简单降级为模糊的 `RUNTIME_ERROR`.

如后续为本组 editor 命令新增专用错误码,必须继续遵守全项目“多真源同步”护栏:

1. 错误码常量真源
2. 错误码白名单真源
3. 对外导出文档真源(例如技能文档中的错误码清单)

## 4. 各命令需求

### 4.1 `editor.getState`

#### 目标

提供一个轻量,稳定,适合频繁调用的编辑器状态快照.

#### 最小输入要求

- `params` 必须为 `{}`.
- 不接受与本命令无关的额外业务字段.

#### 必须回答的信息

至少包括:

- 当前是否处于 Play Mode.
- 当前是否正在进入或退出 Play Mode.
- 当前是否暂停.
- 当前是否正在编译.
- 当前是否正在刷新资源.
- 当前 active scene 的基础信息.

#### 结果要求

- 结果应偏轻量,适合在复杂操作前先调用一次.
- 不应返回过大的层级或组件详情.
- 结果字段命名必须稳定,避免调用方每次猜测.

#### 典型用途

- 在执行 `scene.*` 编辑前确认是否处于 Edit Mode.
- 在触发菜单前确认当前是否仍在编译或刷新.
- 在准备执行回退前确认当前状态是否允许继续操作.

### 4.2 `editor.getContext`

#### 目标

提供一个比 `editor.getState` 更完整的编辑器上下文快照,帮助 AI 理解当前编辑器工作面.

这里的"编辑器上下文"可以简单理解为: 当前大概在看什么,选中了什么,当前工作焦点大概在哪个区域.

#### 最小输入要求

- `params` 必须为 `{}`.
- 不接受与本命令无关的额外业务字段.

#### 必须覆盖的上下文方向

至少包括:

- 当前 selection 摘要.
- 当前 active scene 摘要.
- 当前已加载场景摘要.
- 当前 Project/Hierarchy 相关上下文的基础信息.
- 当前是否有打开对象或资源的上下文线索.

#### 边界

- `editor.getContext` 不是用来替代 `scene.queryHierarchy` 或 `prefab.queryComponents` 的.
- 它返回的是"上下文摘要",不是某一类资源的完整详情.
- 当需要深入查看场景层级,组件属性,Prefab 结构时,仍应调用相应专用技能.
- 它应该返回"当前看起来在处理什么"的摘要,而不是返回完整层级树,完整组件明细,完整资源列表.

#### 设计要求

- 要让非专业人士也能看懂当前编辑器大概处在什么工作面.
- 要能帮助 AI 决定下一步是走 `scene.*`,`prefab.*`,`playmode.*` 还是 `editor.runAction`.

### 4.3 `editor.select`

#### 目标

让 AI 可以显式设置当前编辑器选中对象,为后续菜单执行,工具调用,人工接力检查等场景提供桥梁.

#### 最小输入要求

`params.targetKind` 必填,首版冻结为以下两类:

1. `sceneGameObject`
2. `projectAsset`

当 `targetKind=sceneGameObject` 时,至少要求:

- `sceneName`: 必填.
- `objectPath`: 必填.
- `siblingIndex`: 必填.

当 `targetKind=projectAsset` 时,至少要求:

- `assetPath`: 必填,且采用 `Assets/...` 路径口径.

首版选择语义冻结如下:

- 首版仅支持单目标选择,不支持一次命令中传入多选目标.
- 对于场景对象,不允许省略 `sceneName` 后让系统猜测目标场景.
- 对于资源选择,首版允许常规资源路径选择; 若后续要细分文件夹,场景资源,子资源等差异,需另行补文档.

#### 支持的选择目标

首版至少应支持:

- 选择场景中的 GameObject.
- 选择 Project 中的资源.

#### 关键要求

- 必须明确返回本次实际选中了什么.
- 当传入的目标不存在时,不能静默成功.
- 当有同名对象时,必须有稳定定位规则,不能只按名称猜测.
- 当调用方给出的定位信息不足以唯一确定目标时,必须返回 `AMBIGUOUS_TARGET`,而不是由系统擅自猜测.

#### 与现有能力关系

- 场景对象定位应与 `scene.*` 的 `sceneName + objectPath + siblingIndex` 口径保持对齐.
- 资源定位应与现有 `Assets/...` 路径口径保持一致.

### 4.4 `editor.getSelection`

#### 目标

返回当前编辑器已选中的对象或资源摘要.

#### 最小输入要求

- `params` 必须为 `{}`.
- 不接受与本命令无关的额外业务字段.

#### 结果要求

至少应返回:

- 选中数量.
- 每个选中项的类型摘要.
- 若是场景对象,应尽量返回可用于后续 `scene.*` 的定位信息.
- 若是资源,应尽量返回可用于后续查看或编辑的资源路径.

#### 设计目的

该命令应成为 `editor.select` 的自然验收命令.

也就是说,推荐闭环是:

1. `editor.select`
2. `editor.getSelection`
3. 再执行 `editor.executeMenu` 或 `editor.runAction`

### 4.5 `editor.undo`

#### 目标

提供编辑器级撤销能力,让 AI 在判断失误时可以回退最近一步或最近几步操作.

#### 最小输入要求

- `params.steps`: 必填,int,表示要撤销的步数.
- `steps` 必须 >= 1.
- 首版建议调用方优先从 `steps=1` 开始使用,避免一次性大步回退造成理解偏差.

#### 必须满足的需求

- 调用结果必须明确告诉调用方是否真的撤销了内容.
- 当没有可撤销内容时,结果必须清晰可读,不能让调用方误以为已回退.
- 结果中应尽量包含最基本的撤销摘要,例如是否成功执行撤销,当前是否还有可继续撤销内容.
- 当请求的 `steps` 大于当前可撤销步数时,结果必须清晰说明实际撤销了几步,不能让调用方误以为请求的步数全部生效.

#### 关键边界

- 本命令是编辑器级回退能力,不是对 `prefab.batchEdit` 或 `scene.batchEdit` 事务语义的替代.
- 本命令只负责 Unity 编辑器当前 Undo 栈上的回退,不承诺跨 Editor 重启持久保存.

### 4.6 `editor.redo`

#### 目标

提供编辑器级重做能力,与 `editor.undo` 配对使用.

#### 最小输入要求

- `params.steps`: 必填,int,表示要重做的步数.
- `steps` 必须 >= 1.
- 首版建议调用方优先从 `steps=1` 开始使用.

#### 必须满足的需求

- 结果必须明确告诉调用方是否真的重做了内容.
- 当没有可重做内容时,必须给出清晰结果.
- 与 `editor.undo` 的结果结构应尽量同构,避免调用方分两套解析逻辑.
- 当请求的 `steps` 大于当前可重做步数时,结果必须清晰说明实际重做了几步.

### 4.7 `editor.executeMenu`

#### 目标

允许 AI 按菜单路径执行 Unity Editor 菜单项,用于复用现有编辑器生态中大量基于菜单的能力.

#### 最小输入要求

- `params.menuPath`: 必填,string.
- `menuPath` 应使用清晰的菜单路径表达,例如 `Tools/Export/Build Cards`.
- 不要求调用方提供反射签名或完整类名.

#### 关键要求

- 命令必须明确返回菜单是否真的被执行.
- 当菜单不存在时,必须返回 `MENU_ITEM_NOT_FOUND`.
- 当菜单存在但当前上下文下不可执行时,必须返回 `MENU_ITEM_NOT_EXECUTABLE`.
- 对高风险菜单仍必须允许安全拦截,命中时必须返回 `FORBIDDEN_EDITOR_ACTION`,不能因为是菜单路径就放行.

#### 与 `editor.runAction` 的边界

- 菜单级稳定入口优先用 `editor.executeMenu`.
- 项目私有工具方法,且没有稳定菜单语义的,继续用 `editor.runAction`.
- 调用方已经知道稳定菜单路径时,不应再被要求绕到 `editor.runAction`.

### 4.8 `editor.getTags`

#### 目标

返回当前项目可用的标签列表.

#### 最小输入要求

- `params` 必须为 `{}`.

#### 设计要求

- 结果应尽量简洁,直接返回可用于后续判断和填写的 tags.
- 当项目没有自定义 tags 时,也应返回稳定结构,而不是报错.
- 返回结果应包含 Unity 当前可用的 tags,包括默认项与项目新增项.
- 返回顺序应稳定,避免同一项目多次调用结果顺序飘动.

#### 典型用途

- 修改对象前先确认 tag 是否存在.
- 让 AI 减少靠猜测填写 tag 名称.

### 4.9 `editor.getLayers`

#### 目标

返回当前项目可用的层列表.

#### 最小输入要求

- `params` 必须为 `{}`.

#### 设计要求

- 结果应尽量同时体现层编号和层名称,便于后续使用.
- 对空位或未命名层位应有清晰表达,避免误导调用方.
- 首版应按 layer index 升序返回.
- 首版建议固定覆盖 0-31 全部槽位,即使某些槽位未命名,也应以稳定方式表达,不要有时省略,有时返回.

#### 典型用途

- 修改对象 layer 前先确认候选列表.
- 帮助 AI 理解项目当前的层配置.

### 4.10 `editor.setPauseOnError`

#### 目标

允许 AI 显式控制 Unity Editor 的 Pause On Error 开关,以便在调试 Play Mode 问题时保留现场.

#### 最小输入要求

- `params.enabled`: 必填,bool.
- 应采用非常直观的布尔语义,例如 `enabled=true/false`.

#### 结果要求

- 必须明确返回修改前状态与修改后状态.
- 当重复设置为相同值时,建议仍返回 success,但要明确说明本次没有实际变化.

#### 边界

- 本命令控制的是编辑器调试开关,不是进入或退出暂停状态本身.
- 它不替代 `playmode.start`,`playmode.stop`,也不替代未来可能出现的 `editor.pause` 命令.

## 5. 推荐工作流与闭环

### 5.1 编辑器上下文判断闭环

推荐顺序:

1. `editor.getState`
2. `editor.getContext`
3. 再决定进入 `scene.*`,`prefab.*`,`playmode.*` 或 `editor.runAction`

目的:

- 减少盲目执行.
- 先看当前编辑器是否适合下一步动作.

### 5.2 选择后执行菜单闭环

推荐顺序:

1. `editor.select`
2. `editor.getSelection`
3. `editor.executeMenu`
4. 必要时再用 `log.query` 或专用查看技能验证

目的:

- 让选择动作可被确认.
- 避免对当前到底选中了什么产生误判.

### 5.3 修改失败后的回退闭环

推荐顺序:

1. 执行编辑器动作或菜单
2. 若结果不符合预期,调用 `editor.undo`
3. 必要时调用 `editor.redo` 做对照验证
4. 再继续下一步

目的:

- 为复杂自动化提供最低限度的全局回退能力.

## 6. 举例覆盖需求和边缘情况

### 例 1: 在执行场景编辑前先确认编辑器状态

输入示例:

```json
{
  "batchId": "batch_editor_state_001",
  "commands": [
    {
      "id": "cmd_state",
      "type": "editor.getState",
```

## 7. 与其他技能的协同要求

### 7.1 与 `unity-editor-action` 的协同

- `unity-editor-action` 继续承载 `editor.runAction`.
- 新 editor 控制命令建议形成独立 skill 或在后续整理时归为同一组编辑器控制能力导出.
- 在技能文档层必须清楚写明: 通用稳定控制优先用显式 `editor.*`,项目私有工具动作优先用 `editor.runAction`.

### 7.2 与 `unity-scene-view` 和 `unity-scene-edit` 的协同

- `editor.getContext` 只给上下文摘要,不替代场景专用查看命令.
- `editor.select` 若选择的是场景对象,结果应尽量与 `scene.*` 定位口径对齐.

### 7.3 与 `unity-log` 的协同

- 执行 `editor.executeMenu` 或 `editor.runAction` 后,若需要确认副作用,可结合 `log.query` 和 `log.refresh` 验证.
- 当编辑器动作触发资源刷新或重新编译时,推荐调用方显式使用 `log.refresh` 等待闭环完成.

## 8. 验收理解口径

以下情况视为本需求被正确实现:

- 非专业人士能通过文档理解每个 `editor.*` 命令是做什么的.
- 调用方能清楚区分 `editor.runAction` 与显式 `editor.*` 命令的边界.
- 调用方能在复杂编辑器操作前先读取状态和上下文,而不是盲目执行.
- 调用方能在依赖 selection 的工作流中稳定完成"选择 -> 验证 -> 执行".
- 调用方能在操作不符合预期时使用 `undo/redo` 做最低限度的全局回退.
- 调用方能减少对 tags, layers,当前状态的猜测式试错.
