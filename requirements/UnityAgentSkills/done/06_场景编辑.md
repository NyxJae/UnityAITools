# 需求文档 - 场景编辑功能

## 1. 项目现状与核心目标

### 1.1 用户需求简述

需要为场景提供一套完整,稳定,对非专业人士也易理解的编辑能力,用于在 Unity Editor 中编辑已存在的 scene asset 内容.

本需求与 `05_场景查看.md` 配套使用. 查看与编辑分开维护,共同构成完整的场景工作流.

编辑前必须先使用场景查看能力获取 `sceneName`,`objectPath`,`siblingIndex`,`componentIndex` 等定位信息.

### 1.2 技能边界

- `unity-scene-view`: 只负责查看,包含 `scene.open`,`scene.queryHierarchy`,`scene.queryComponents`.
- 场景编辑能力负责 scene asset 编辑,并承载与 prefab asset / prefab instance 关系相关的桥接落位与还原能力需求.
- `scene.*` 只负责场景对象本体编辑,不包含查看命令.
- `prefabBridge.*` 仍是独立桥接命令组,但需求文档不再单独成篇.
- 所有编辑都只针对已加载且可保存的 scene asset 内容.
- 所有场景编辑命令都仅允许在编辑模式执行,Play 模式一律报错.

### 1.3 核心目标

- 让外部工具可以仅通过查看命令 + 场景编辑命令完成常见场景编辑闭环.
- 支持对场景内 GameObject 进行属性,层级,排序,改名,布局等编辑.
- 支持对组件进行增删改,并支持常见引用类型写入.
- 支持一次编辑多个步骤,并在需要时保证严格事务语义.
- 在多场景已加载情况下,仍能通过 `sceneName + objectPath + siblingIndex` 稳定定位目标.
- 返回清晰,可验证,对非专业人士友好的结果信息.

## 2. 范围与边界

### 2.1 功能点简述

本需求覆盖以下 11 个编辑命令:

#### A. 基础属性编辑

- [ ] `scene.setGameObjectProperties`: 修改 GameObject 基础属性.
- [ ] `scene.renameGameObject`: 仅修改 GameObject 名称.

#### B. 层级结构编辑

- [ ] `scene.createGameObject`: 在指定父节点下创建 GameObject.
- [ ] `scene.deleteGameObject`: 删除指定 GameObject,并级联删除其子物体.
- [ ] `scene.moveOrCopyGameObject`: 移动或复制 GameObject 到新的父节点.
- [ ] `scene.setSiblingIndex`: 仅调整同一父节点下的排序.

#### C. 组件编辑

- [ ] `scene.addComponent`: 给指定 GameObject 添加组件.
- [ ] `scene.setComponentProperties`: 修改指定组件属性.
- [ ] `scene.deleteComponent`: 删除指定组件.

#### D. 布局编辑

- [ ] `scene.setTransform`: 批量修改 Transform 本地坐标相关字段.
- [ ] `scene.setRectTransform`: 批量修改 RectTransform UI 布局字段.

#### E. 批量事务编辑

- [ ] `scene.batchEdit`: 在一次加载同一个场景编辑上下文的前提下,串行执行多个编辑子操作,最后统一保存.

### 2.2 排除项

- 不包含场景查看命令,查看能力由 `05_场景查看.md` 单独负责.
- 不支持在 Play 模式下执行任何场景编辑命令.
- 不支持省略 `sceneName` 依赖 activeScene 猜测目标场景.
- 不支持跨多个场景执行一次 `scene.batchEdit`.
- 不支持通过 `instanceID` 直接编辑对象或组件.
- 不支持创建新 scene 资源文件,只支持编辑已存在场景.
- 不在本需求中定义“切换 activeScene”的额外命令.

## 3. 通用协议与一致性要求

### 3.1 前置依赖与工作流

所有编辑命令都默认依赖查看能力:

1. 先用 `scene.open` 打开目标场景,或确认目标场景已加载.
2. 用 `scene.queryHierarchy` 获取 `sceneName`,`objectPath`,`siblingIndex`.
3. 如涉及组件,再用 `scene.queryComponents` 获取 `componentType`,`componentIndex` 等信息.
4. 再调用具体编辑命令.
5. 编辑后可再次查询验证结果.

### 3.2 场景与对象定位规则

- 所有编辑命令都必须显式传入 `sceneName`.
- `sceneName` 必须与 Unity `Scene.name` 大小写一致.
- GameObject 定位统一使用 `sceneName + objectPath + siblingIndex`.
  - `objectPath`: 例如 `Main/Panel_Content/ConfirmButton`.
  - `siblingIndex`: 同一路径下同名对象索引,从 0 开始,默认 0.
- 父节点定位统一使用 `sceneName + parentPath + parentSiblingIndex`.
- 组件定位统一使用 `componentType + componentIndex`.
  - `componentIndex`: 同类型组件索引,从 0 开始.
- 位置相关字段语义必须区分:
  - `siblingIndex`: 仅用于同名对象定位.
  - `insertSiblingIndex`: 创建时的插入位置.
  - `targetSiblingIndex`: 移动或复制后的目标位置.
  - `newSiblingIndex`: 仅用于重排序.

### 3.3 模式约束与保存语义

- 所有 `scene.*` 编辑命令都仅允许在编辑模式执行.
- Play 模式调用任意场景编辑命令时:
  - `error.code = ONLY_ALLOWED_IN_EDIT_MODE`.
  - `error.message` 需对非专业人士可读,例如"当前为 Play 模式,仅编辑模式可编辑场景".
  - `error.detail` 必须包含 `OnlyAllowedInEditMode` 关键字.
- 单命令编辑默认在成功后自动保存目标场景.
- `scene.batchEdit` 只能在同一个 `sceneName` 上工作.
- `scene.batchEdit` 必须在同一个场景编辑上下文中串行执行多个子操作.
- 只要存在任一子操作失败,则整体 `saved=false`,且必须保证该场景文件未发生变化.

### 3.4 错误码与结果风格

- 错误码必须遵循框架白名单机制.
- 若新增场景编辑专用错误码,必须同步到错误码常量真源,框架白名单,以及对应 SkillsExporter 文档真源,避免对外被降级为 `RUNTIME_ERROR`.
- 调用方应优先依赖 `error.code` 判断失败原因,不要依赖 `error.message` 的固定文本.
- 成功结果应尽量回显:
  - 输入的 `sceneName`
  - 目标对象路径或组件信息
  - 修改后的关键字段
  - 是否保存成功

### 3.5 prefab 与 scene 桥接落位约定

- `prefabBridge.*` 仍保留为独立命令组,但其与 scene 中实例落位,识别,还原,解包有关的需求统一并入本文件维护.
- `prefabBridge.instantiateInScene`,`prefabBridge.getInstanceSource`,`prefabBridge.getInstanceRelationship`,`prefabBridge.revertOverrides`,`prefabBridge.unpackInstance` 都必须显式传入 `sceneName`.
- `prefabBridge.instantiateInScene` 必须要求 `prefabPath` 为 `Assets/.../*.prefab`,且只允许向已加载且可保存的 scene asset 实例化. 成功后必须保存目标 scene.
- `instantiateInScene` 中允许的 `name`,`localPosition`,`localRotation`,`localScale`,`anchorMin`,`anchorMax`,`anchoredPosition`,`sizeDelta`,仅表示实例化当次的初始落位与命名,不扩展为通用 scene 编辑入口.
- `prefabBridge.getInstanceSource` 若目标不是 prefab instance,必须成功返回 `isPrefabInstance=false`,不得把普通 scene 对象误报为错误.
- `prefabBridge.getInstanceRelationship` 必须能说明 regular,variant,nested 等关系,并补充非专业人士可读的 `relationshipSummary`.
- `prefabBridge.revertOverrides` 与 `prefabBridge.unpackInstance` 成功后都必须保存目标 scene.
- `revertOverrides(selected)` 必须通过 `overrideIds[]` 精确指定要还原的改动,并返回 `revertedOverrideIds[]`,`hasOverridesBefore`,`hasOverridesAfter`.
- `unpackInstance` 至少支持 `outermost` 与 `completely` 两种模式,且返回必须明确对象已不再是 prefab instance.
- 所有桥接命令在 Play Mode 下都必须稳定返回 `ONLY_ALLOWED_IN_EDIT_MODE`,文案要明确说明桥接操作仅允许在编辑模式执行.
- 桥接层不再定义专用 `*Batch` 命令. 批量实例化,批量 revert,批量 unpack 必须复用框架级 `batchId + commands[]` 串行机制.

## 4. 各命令需求

### 4.1 基础属性编辑

#### 4.1.1 `scene.setGameObjectProperties`

**用途**: 修改 GameObject 的基础属性.

**可修改范围**:

- `name`
- `tag`
- `layer`
- `isActive`
- `isStatic`
- `hideFlags`
- `navMeshLayer`
- `icon`

**要求**:

- [ ] 接收 `sceneName`,`objectPath`,`siblingIndex`.
- [ ] 只修改输入中明确提供的字段,未提供字段保持不变.
- [ ] 修改后自动保存场景.
- [ ] 返回修改后的对象关键信息.

#### 4.1.2 `scene.renameGameObject`

**用途**: 仅修改 GameObject 名称.

**要求**:

- [ ] 接收 `sceneName`,`objectPath`,`siblingIndex`,`newName`.
- [ ] 仅改名,不改变父子关系.
- [ ] 允许同级重名,重名治理由调用方负责.
- [ ] 当 `siblingIndex < 0` 时返回 `INVALID_FIELDS`.
- [ ] 修改后自动保存场景.
- [ ] 返回 `oldName`,`newName`,`oldPath`,`newPath`,`saved`.

### 4.2 层级结构编辑

#### 4.2.1 `scene.createGameObject`

**用途**: 在指定父节点下创建新 GameObject.

**要求**:

- [ ] 接收 `sceneName`,`parentPath`,`parentSiblingIndex`.
- [ ] 支持 `insertSiblingIndex`,不传时默认插入末尾.
- [ ] `insertSiblingIndex` 超出范围时自动修正到有效范围.
- [ ] 支持设置基础属性: `name`,`tag`,`layer`,`isActive`,`isStatic`,`hideFlags`.
- [ ] 修改后自动保存场景,并返回新对象路径与索引.

#### 4.2.2 `scene.deleteGameObject`

**用途**: 删除指定 GameObject 及其所有子物体.

**要求**:

- [ ] 支持按 `sceneName + objectPath + siblingIndex` 定位目标对象.
- [ ] 删除时进行级联删除.
- [ ] 不允许删除整个场景本身,删除对象不影响其他已加载场景.
- [ ] 自动保存场景.
- [ ] 返回删除对象的关键信息.

#### 4.2.3 `scene.moveOrCopyGameObject`

**用途**: 将对象移动或复制到新的父节点下.

**要求**:

- [ ] 顶层必须显式传入 `sceneName`.
- [ ] 通过 `objectPath + siblingIndex` 定位源对象,通过 `parentPath + parentSiblingIndex` 定位目标父节点.
- [ ] 源对象与目标父节点必须属于同一 `sceneName`.
- [ ] 用参数明确区分“移动”与“复制”.
- [ ] 支持 `targetSiblingIndex` 指定目标位置.
- [ ] `targetSiblingIndex` 超出范围时自动修正到有效范围.
- [ ] 默认仅允许在同一 `sceneName` 内移动或复制,不允许跨场景改父节点.
- [ ] 自动保存场景.
- [ ] 返回操作类型,变更前路径,变更后路径,变更前父节点,变更后父节点,实际应用的位置.

#### 4.2.4 `scene.setSiblingIndex`

**用途**: 仅调整同一父节点下的排序.

**要求**:

- [ ] 接收 `sceneName`,`objectPath`,`siblingIndex`,`newSiblingIndex`.
- [ ] 不改变父子关系.
- [ ] 当 `newSiblingIndex < 0` 时返回 `INVALID_FIELDS`.
- [ ] 当 `newSiblingIndex` 超出范围时自动修正到有效范围,默认行为为移动到末尾.
- [ ] 自动保存场景.
- [ ] 返回 `oldSiblingIndex`,`newSiblingIndexApplied`,`saved`.

### 4.3 组件编辑

#### 4.3.1 `scene.addComponent`

**用途**: 给指定对象添加组件.

**要求**:

- [ ] 接收 `sceneName`,`objectPath`,`siblingIndex`,`componentType`,并按 `sceneName + objectPath + siblingIndex` 定位目标对象后添加组件.
- [ ] 支持 Unity 内置组件和自定义组件.
- [ ] `componentType` 支持全名或短名.
- [ ] 对不可重复组件,重复添加时返回明确错误.
- [ ] 自动保存场景.
- [ ] 返回 `componentType`,`componentIndex`,`componentInstanceID`,`saved`.

#### 4.3.2 `scene.setComponentProperties`

**用途**: 修改指定组件的序列化属性.

**要求**:

- [ ] 通过 `sceneName + objectPath + siblingIndex + componentType + componentIndex` 精确定位组件.
- [ ] 支持简单类型,枚举,数组,嵌套对象.
- [ ] 支持引用类型写入:
  - `null`
  - 同场景内 GameObject 引用
  - 同场景内组件引用
  - 项目资源引用(`assetPath`)
- [ ] 自动保存场景.
- [ ] 返回 `modifiedProperties` 等变更明细.
- [ ] 当未提供任何可修改属性时,应返回明确错误,避免 silent success.

#### 4.3.3 `scene.deleteComponent`

**用途**: 删除指定组件.

**要求**:

- [ ] 通过 `componentType + componentIndex` 精确定位组件.
- [ ] 支持同类型多组件按索引删除.
- [ ] 必需组件(例如 Transform)不可删除.
- [ ] 自动保存场景.
- [ ] 返回 `deletedComponentType`,`deletedComponentIndex`,`deletedComponentInstanceID`,`saved`.

### 4.4 布局编辑

#### 4.4.1 `scene.setTransform`

**用途**: 批量修改 Transform 的本地坐标相关字段.

**可写字段**:

- `localPosition`(x,y,z)
- `localRotationEuler`(x,y,z)
- `localScale`(x,y,z)

**要求**:

- [ ] 接收 `sceneName`,`objectPath`,`siblingIndex`.
- [ ] 只修改输入中明确提供的字段.
- [ ] 未提供任何可写字段时,返回 `INVALID_FIELDS`.
- [ ] 自动保存场景.
- [ ] 返回 `modifiedFields` 和修改后值.

#### 4.4.2 `scene.setRectTransform`

**用途**: 批量修改 RectTransform 的 UI 布局字段.

**可写字段**:

- `anchorMin`(x,y)
- `anchorMax`(x,y)
- `pivot`(x,y)
- `anchoredPosition`(x,y)
- `sizeDelta`(x,y)
- `offsetMin`(x,y)
- `offsetMax`(x,y)

**要求**:

- [ ] 目标对象必须有 `RectTransform`,否则返回 `COMPONENT_NOT_FOUND`.
- [ ] 修改前后必须满足 `anchorMin.x <= anchorMax.x` 与 `anchorMin.y <= anchorMax.y`.
- [ ] 未提供任何可写字段时,返回 `INVALID_FIELDS`.
- [ ] 自动保存场景.
- [ ] 返回 `modifiedFields` 和修改后值.

### 4.5 批量事务编辑

#### 4.5.1 `scene.batchEdit`

**用途**: 在同一个场景编辑上下文中串行执行多个子操作,最后统一保存.

**允许包含的子操作**:

- `scene.setGameObjectProperties`
- `scene.renameGameObject`
- `scene.createGameObject`
- `scene.deleteGameObject`
- `scene.moveOrCopyGameObject`
- `scene.setSiblingIndex`
- `scene.addComponent`
- `scene.setComponentProperties`
- `scene.deleteComponent`
- `scene.setTransform`
- `scene.setRectTransform`

**要求**:

- [ ] 顶层参数必须显式传入 `sceneName`.
- [ ] `operations[].params` 中禁止再次出现 `sceneName`.
- [ ] 子操作按输入顺序严格串行执行.
- [ ] `mode=stopOnError` 时,后续未执行子操作返回 `SKIPPED`.
- [ ] `mode=continueOnError` 时,可以继续执行后续子操作,但只要任一子操作失败,整体仍不得保存.
- [ ] 子操作在 batch 内不得隐式保存.
- [ ] 结果中必须返回 `operationResults[]` 与整体 `saved`,并与 `prefab.batchEdit` 保持一致的字段风格.

## 5. 举例覆盖需求和边缘情况

### 5.1 例 1: 先查看再改名

1. 先使用 `scene.queryHierarchy` 找到目标对象所在的 `sceneName`,`objectPath`,`siblingIndex`.
2. 再调用 `scene.renameGameObject` 改名.
3. 再次查询层级,确认 `newPath` 正确.

### 5.2 例 2: 在多场景已加载时精确编辑

假设当前已同时加载 `Main` 和 `Battle` 两个场景,且两个场景里都有 `Canvas/Panel/ConfirmButton`.

- 调用方必须显式指定 `sceneName="Main"` 或 `sceneName="Battle"`.
- 不允许只传 `objectPath` 让系统猜测.
- 成功后只保存被编辑的那个场景.

### 5.3 例 3: 创建新按钮并补齐组件

1. 使用 `scene.createGameObject` 在 `Main/Panel_Content` 下创建 `UIButton_New`.
2. 使用 `scene.addComponent` 添加 `UnityEngine.UI.Button`.
3. 使用 `scene.setRectTransform` 设置布局.
4. 使用 `scene.setComponentProperties` 设置按钮组件属性.

### 5.4 例 4: 同名对象的精确定位

假设 `Main/Panel_Content` 下有两个同名 `UIButton_Confirm`:

- 第一个对象使用 `sceneName="Main", objectPath="Main/Panel_Content/UIButton_Confirm", siblingIndex=0`
- 第二个对象使用 `sceneName="Main", objectPath="Main/Panel_Content/UIButton_Confirm", siblingIndex=1`

调用方不得只靠路径字符串猜测目标对象.

### 5.5 例 5: 同类型多组件的精确定位

假设同一个对象上挂了两个同类型组件:

- 先用 `scene.queryComponents` 查询组件列表.
- 再通过 `componentType + componentIndex` 精确指定目标组件.
- 例如,删除第二个同类型组件时,使用 `componentIndex=1`.

### 5.6 例 6: 只调整排序,不移动父节点

当只是想修改同一父节点下的显示顺序时:

- 应使用 `scene.setSiblingIndex`.
- 不应把纯排序操作误写成 `scene.moveOrCopyGameObject`.

### 5.7 例 7: RectTransform 校验失败

如果输入导致 `anchorMin.x > anchorMax.x` 或 `anchorMin.y > anchorMax.y`:

- 命令必须失败.
- 不应出现部分字段已写入的情况.
- 应返回非专业人士可读的错误提示.

### 5.8 例 8: Play 模式下尝试编辑场景

如果在 Play 模式调用 `scene.setTransform` 或其他任意场景编辑命令:

- 命令必须失败.
- 返回 `error.code = ONLY_ALLOWED_IN_EDIT_MODE`.
- 不自动退出 Play 模式.
- 不对场景文件产生任何修改.

### 5.9 例 9: batchEdit 严格事务

当一次 `scene.batchEdit` 中包含 3 个子操作:

1. 创建节点成功.
2. 设置布局失败.
3. 改名未执行或继续执行,由 `mode` 决定.

最终要求:

- 整体 `saved=false`.
- scene 文件必须保持未修改状态.
- 调用方可通过逐步结果知道哪一步失败.

### 5.10 例 10: 不允许跨场景移动对象

假设对象当前位于 `Main` 场景,目标父节点位于 `Battle` 场景:

- `scene.moveOrCopyGameObject` 必须失败.
- 错误提示要明确说明当前版本只允许在同一 `sceneName` 内编辑层级关系.
- 不得出现对象已从原场景删除,但未正确进入目标场景的半成功状态.

### 5.11 例 11: 将 prefab 实例化到指定场景

目标: 把 `Assets/Resources/Prefabs/ButtonConfirm.prefab` 放到 `Main` 场景的 `Canvas/Panel` 下.

- 调用 `prefabBridge.instantiateInScene` 后,必须成功创建 prefab instance.
- 返回中必须包含 `sceneName`,`prefabPath`,`instancePath`,`siblingIndex`,`isPrefabInstance=true`.
- 目标 scene 必须已保存.

### 5.12 例 12: 查询对象是否仍与 prefab 关联

目标: 查询 `Main/TempDebugRoot` 的 prefab 来源.

- 如果目标是普通 scene 对象,命令也必须成功返回.
- 返回中必须明确 `isPrefabInstance=false`.
- 不应把“普通对象”误报为错误.

### 5.13 例 13: 只还原部分 override

假设一个实例同时改了位置和按钮文本,现在只想还原位置:

- 调用 `prefabBridge.revertOverrides(revertMode=selected, overrideIds=[...])` 后,只还原被选中的改动.
- 返回必须明确 `revertedOverrideIds[]`.
- 文本 override 仍保留,并能从结果中看出来.

### 5.14 例 14: 解包 nested prefab

假设一个复杂 UI prefab 内部还嵌套了另一个 prefab:

- `prefabBridge.unpackInstance(unpackMode=outermost)` 只解除最外层关联.
- `prefabBridge.unpackInstance(unpackMode=completely)` 尽可能解除整条嵌套关系.
- 返回必须让非专业人士看懂对象是否还属于 prefab instance.

### 5.15 例 15: 通过框架级 commands[] 批量实例化与还原

假设同一批次中:

1. 第 1 个命令把 `Button.prefab` 实例化到 `Main` 场景.
2. 第 2 个命令对 `Battle` 场景中的某个实例执行 `prefabBridge.revertOverrides`.
3. 第 3 个命令因为 `sceneName` 未加载而失败.

则要求:

- 各命令结果必须按框架级 `results[]` 逐项返回.
- 成功的命令各自保存对应 scene.
- 失败命令不得伪装为成功保存.

## 7. 验收口径

- [ ] 以上 11 个场景编辑命令均有清晰且一致的输入输出语义.
- [ ] 调用方可仅通过场景查看命令 + 场景编辑命令完成常见场景编辑闭环.
- [ ] 所有编辑命令都强制要求 `sceneName`,多场景下不存在歧义定位.
- [ ] `scene.addComponent`,`scene.setComponentProperties`,`scene.deleteComponent` 都能基于统一定位规则稳定工作.
- [ ] `scene.moveOrCopyGameObject` 只允许在同一 `sceneName` 内编辑层级关系,不得跨场景半成功.
- [ ] 同名对象可通过 `sceneName + objectPath + siblingIndex` 稳定定位.
- [ ] 同类型多组件可通过 `componentType + componentIndex` 稳定定位.
- [ ] 所有场景编辑命令仅允许在编辑模式执行,Play 模式下稳定报错且不产生修改.
- [ ] `scene.batchEdit` 返回稳定的 `operationResults[]`,并满足严格事务语义,失败时不产生部分落盘.
- [ ] 能将指定 prefab asset 实例化到指定已加载 scene 中,并返回稳定定位信息.
- [ ] 能识别 scene 中对象是否为 prefab instance,并在普通对象场景下稳定返回 `isPrefabInstance=false`.
- [ ] 能将全部或部分 overrides revert 回 prefab 默认值,并明确返回被还原的 `overrideIds[]`.
- [ ] 能对 prefab instance 执行 unpack,并清楚区分 `outermost` 与 `completely` 两种模式.
- [ ] 所有桥接命令在 Play Mode 下都稳定报错,且桥接文案对非专业人士可理解.
- [ ] 桥接层不再定义专用 `*Batch` 命令,批量实例化,批量 revert,批量 unpack 统一复用框架级 `batchId + commands[]`.
- [ ] 若新增场景编辑专用错误码,对应实现与导出文档都已同步更新,不会被降级为 `RUNTIME_ERROR`.
- [ ] 文档对非专业人士可读,且足以支撑技能说明与验收对齐.
