# 需求文档 - 预制体编辑功能

## 1. 项目现状与核心目标

### 1.1 用户需求简述

需要为 `unity-prefab-edit` 提供一套完整,稳定,对非专业人士也易理解的预制体编辑能力,用于在 Unity Editor 中编辑已有 prefab asset 文件.

本需求是 `04_预制体编辑01.md`,`04_预制体编辑02.md`,`04_预制体编辑03.md`,`04_预制体编辑04.md` 的最终合并版. 合并后,预制体编辑只保留这一份需求文档作为唯一真源.

编辑前必须先使用 `unity-prefab-view` 获取 `rootPath`,`objectPath`,`siblingIndex`,`componentIndex` 等定位信息.

### 1.2 技能边界

- `unity-prefab-view`: 只负责查看,包含 `prefab.queryHierarchy`,`prefab.queryComponents`.
- `unity-prefab-edit`: 负责 prefab asset 编辑,并承载与 scene 中 prefab instance 相关的桥接回写能力需求.
- 编辑类命令统一通过 `UnityAgentSkills.Plugins.Prefab.PrefabCommandsPlugin` 提供.
- `prefab.*` 只针对 prefab asset 文件本体编辑.
- `prefabBridge.*` 仍是独立桥接命令组,但需求文档不再单独成篇.
- 直接修改 scene 中对象结构或组件不属于本文件,应由 `06_场景编辑.md` 承接.

### 1.3 核心目标

- 让外部工具可以可靠完成预制体的常见编辑闭环.
- 支持对 GameObject 进行属性,层级,排序,改名,布局等编辑.
- 支持对组件进行增删改,并支持常见引用类型写入.
- 支持一次编辑多个步骤,并在需要时保证严格事务语义.
- 返回清晰,可验证,对非专业人士友好的结果信息.

## 2. 范围与边界

### 2.1 功能点简述

本需求覆盖以下 12 个编辑命令:

#### A. 基础属性编辑

- [ ] `prefab.setGameObjectProperties`: 修改 GameObject 基础属性.
- [ ] `prefab.renameGameObject`: 仅修改 GameObject 名称.

#### B. 层级结构编辑

- [ ] `prefab.createGameObject`: 在指定父节点下创建 GameObject.
- [ ] `prefab.deleteGameObject`: 删除指定 GameObject,并级联删除其子物体.
- [ ] `prefab.moveOrCopyGameObject`: 移动或复制 GameObject 到新的父节点.
- [ ] `prefab.setSiblingIndex`: 仅调整同一父节点下的排序.

#### C. 组件编辑

- [ ] `prefab.addComponent`: 给指定 GameObject 添加组件.
- [ ] `prefab.setComponentProperties`: 修改指定组件属性.
- [ ] `prefab.deleteComponent`: 删除指定组件.

#### D. 布局编辑

- [ ] `prefab.setTransform`: 批量修改 Transform 本地坐标相关字段.
- [ ] `prefab.setRectTransform`: 批量修改 RectTransform UI 布局字段.

#### E. 批量事务编辑

- [ ] `prefab.batchEdit`: 在一次加载同一个 prefab 的前提下,串行执行多个编辑子操作,最后统一保存.

### 2.2 排除项

- 不支持编辑运行时实例,只支持 prefab asset 文件.
- 不支持跨 prefab 编辑,除非某个命令明确说明仅写入资源引用.
- 不支持创建新 prefab 资源文件,只支持编辑已存在 prefab.
- 不支持删除 prefab 根节点.
- 不支持超出 Unity 序列化系统的反射黑魔法写入.
- 不在本需求中定义 prefab 查看命令,查看能力由 `03_预制体查看.md` 单独负责.

## 3. 通用协议与一致性要求

### 3.1 前置依赖与工作流

所有编辑命令都默认依赖查看能力:

1. 先用 `prefab.queryHierarchy` 获取 `rootPath`,`objectPath`,`siblingIndex`.
2. 如涉及组件,再用 `prefab.queryComponents` 获取 `componentType`,`componentIndex` 等信息.
3. 再调用具体编辑命令.
4. 编辑后可再次查询验证结果.

### 3.2 路径与定位规则

- `prefabPath` 必须是工程相对路径,格式为 `Assets/.../*.prefab`.
- GameObject 定位统一使用 `objectPath + siblingIndex`.
  - `objectPath`: 例如 `DialogMain/Panel_Content/ConfirmButton`.
  - `siblingIndex`: 同一路径下同名对象索引,从 0 开始,默认 0.
- 父节点定位统一使用 `parentPath + parentSiblingIndex`.
- 组件定位统一使用 `componentType + componentIndex`.
  - `componentIndex`: 同类型组件索引,从 0 开始.
- 位置相关字段语义必须区分:
  - `siblingIndex`: 仅用于同名对象定位.
  - `insertSiblingIndex`: 创建时的插入位置.
  - `targetSiblingIndex`: 移动或复制后的目标位置.
  - `newSiblingIndex`: 仅用于重排序.

### 3.3 根节点约束

- `prefab.queryHierarchy` 返回的根节点 `path` 视为该 prefab 的 `rootPath`.
- 这里的 `rootPath` 不是一个额外独立对象,而是“根节点自己的路径值”.
- 调用方必须以最近一次查询得到的 `rootPath` 为准,不得自行猜测.
- 不允许删除 prefab 根节点.
- 不允许对根节点执行 `prefab.renameGameObject`,`prefab.setSiblingIndex`,`prefab.setTransform`,`prefab.setRectTransform`.
- 若命中该约束,应返回 `INVALID_FIELDS`,且 `error.message` 必须让非专业人士可读.

### 3.4 错误码与结果风格

- 错误码必须遵循框架白名单机制.
- 调用方应优先依赖 `error.code` 判断失败原因,不要依赖 `error.message` 的固定文本.
- 成功结果应尽量回显:
  - 输入的 `prefabPath`
  - 目标对象路径或组件信息
  - 修改后的关键字段
  - 是否保存成功

### 3.5 批次与保存语义

- 单命令编辑默认在成功后自动保存 prefab.
- `prefab.batchEdit` 只能在同一个 `prefabPath` 上工作.
- `prefab.batchEdit` 必须单次加载 prefab contents,串行执行多个子操作.
- `prefab.batchEdit` 只允许在所有子操作都满足保存条件时统一保存一次.
- 只要存在任一子操作失败,则整体 `saved=false`,且必须保证 prefab asset 文件未发生变化.

### 3.6 prefab 与 scene 桥接回写约定

- `prefabBridge.*` 仍保留为独立命令组,但其与 prefab asset 写回有关的需求统一并入本文件维护.
- `prefabBridge.applyOverrides` 的目标是把 scene 中 prefab instance 的改动写回 prefab asset,不等同于直接编辑 scene 对象.
- `prefabBridge.getOverrides`,`prefabBridge.applyOverrides` 的输入对象定位都必须使用 `sceneName + objectPath + siblingIndex`,不得退化为 `instanceID`.
- `prefabBridge.getOverrides` 返回的每条改动都必须包含稳定可回传的 `overrideId`,供 `applyOverrides(selected)` 复用.
- 当目标 instance 直接来源于普通 prefab 时,`applyOverrides` 默认写回该直接来源 prefab.
- 当目标 instance 直接来源于 prefab variant 时,`applyOverrides` 默认且仅允许写回该直接来源 variant,不得默认越级写回其 base prefab.
- `applyOverrides` 成功后必须保存被写入的 prefab asset,并明确返回 `applyTargetPrefabPath`,`appliedOverrideIds[]`,`hasOverridesBefore`,`hasOverridesAfter`.
- 桥接层不再定义专用 `*Batch` 命令. 批量 apply 必须复用框架级 `batchId + commands[]` 串行机制.
- 桥接结果文案应尽量使用“来源 prefab”“已写回 prefab”“仍保留哪些改动”等非专业人士易懂表述.

## 4. 各命令需求

### 4.1 基础属性编辑

#### 4.1.1 `prefab.setGameObjectProperties`

**用途**: 修改 GameObject 的基础属性.

**可修改范围**:

- `name`
- `tag`
- `layer`
- `isActive`
- `isStatic`
- `hideFlags`
- `navMeshLayer`
- `icon`

**要求**:

- [ ] 只修改输入中明确提供的字段,未提供字段保持不变.
- [ ] 修改后自动保存 prefab.
- [ ] 返回修改后的对象关键信息.
- [ ] 支持在批次中对多个对象分别修改.

#### 4.1.2 `prefab.renameGameObject`

**用途**: 仅修改 GameObject 名称.

**要求**:

- [ ] 接收 `prefabPath`,`objectPath`,`siblingIndex`,`newName`.
- [ ] 仅改名,不改变父子关系.
- [ ] 允许同级重名,重名治理由调用方负责.
- [ ] 当 `siblingIndex < 0` 时返回 `INVALID_FIELDS`.
- [ ] 修改后自动保存 prefab.
- [ ] 返回 `oldName`,`newName`,`oldPath`,`newPath`.

### 4.2 层级结构编辑

#### 4.2.1 `prefab.createGameObject`

**用途**: 在指定父节点下创建新 GameObject.

**要求**:

- [ ] 支持 `parentPath`,`parentSiblingIndex`.
- [ ] 支持 `insertSiblingIndex`,不传时默认插入末尾.
- [ ] `insertSiblingIndex` 超出范围时自动修正到有效范围.
- [ ] 支持设置基础属性: `name`,`tag`,`layer`,`isActive`,`isStatic`,`hideFlags`.
- [ ] 自动保存 prefab,并返回新对象路径与索引.

#### 4.2.2 `prefab.deleteGameObject`

**用途**: 删除指定 GameObject 及其所有子物体.

**要求**:

- [ ] 支持按 `objectPath + siblingIndex` 定位目标对象.
- [ ] 删除时进行级联删除.
- [ ] 不允许删除 prefab 根节点.
- [ ] 自动保存 prefab.
- [ ] 返回删除对象的关键信息.

#### 4.2.3 `prefab.moveOrCopyGameObject`

**用途**: 将对象移动或复制到新的父节点下.

**要求**:

- [ ] 接收源对象路径和目标父节点路径.
- [ ] 用参数明确区分“移动”与“复制”.
- [ ] 支持 `targetSiblingIndex` 指定目标位置.
- [ ] `targetSiblingIndex` 超出范围时自动修正到有效范围.
- [ ] 保持对象及其子物体结构完整.
- [ ] 自动保存 prefab.
- [ ] 返回操作类型,变更前路径,变更后路径,变更前父节点,变更后父节点,实际应用的位置.

#### 4.2.4 `prefab.setSiblingIndex`

**用途**: 仅调整同一父节点下的排序.

**要求**:

- [ ] 接收 `prefabPath`,`objectPath`,`siblingIndex`,`newSiblingIndex`.
- [ ] 不改变父子关系.
- [ ] 当 `newSiblingIndex < 0` 时返回 `INVALID_FIELDS`.
- [ ] 当 `newSiblingIndex` 超出范围时自动修正到有效范围,默认行为为移动到末尾.
- [ ] 自动保存 prefab.
- [ ] 返回 `oldSiblingIndex`,`newSiblingIndexApplied`.

### 4.3 组件编辑

#### 4.3.1 `prefab.addComponent`

**用途**: 给指定对象添加组件.

**要求**:

- [ ] 支持 Unity 内置组件和自定义组件.
- [ ] `componentType` 支持全名或短名.
- [ ] 对不可重复组件,重复添加时返回明确错误.
- [ ] 自动保存 prefab.
- [ ] 返回 `componentType`,`componentIndex`,`componentInstanceID`,`saved`.

#### 4.3.2 `prefab.setComponentProperties`

**用途**: 修改指定组件的序列化属性.

**要求**:

- [ ] 通过 `componentType + componentIndex` 精确定位组件.
- [ ] 支持简单类型,枚举,数组,嵌套对象.
- [ ] 支持引用类型写入:
  - `null`
  - prefab 内 GameObject 引用
  - prefab 内组件引用
  - 项目资源引用(`assetPath`)
- [ ] 自动保存 prefab.
- [ ] 返回 `modifiedProperties` 等变更明细.
- [ ] 当未提供任何可修改属性时,应返回明确错误,避免 silent success.

#### 4.3.3 `prefab.deleteComponent`

**用途**: 删除指定组件.

**要求**:

- [ ] 通过 `componentType + componentIndex` 精确定位组件.
- [ ] 支持同类型多组件按索引删除.
- [ ] 必需组件(例如 Transform)不可删除.
- [ ] 自动保存 prefab.
- [ ] 返回 `deletedComponentType`,`deletedComponentIndex`,`deletedComponentInstanceID`,`saved`.

### 4.4 布局编辑

#### 4.4.1 `prefab.setTransform`

**用途**: 批量修改 Transform 的本地坐标相关字段.

**可写字段**:

- `localPosition`(x,y,z)
- `localRotationEuler`(x,y,z)
- `localScale`(x,y,z)

**要求**:

- [ ] 只修改输入中明确提供的字段.
- [ ] 未提供任何可写字段时,返回 `INVALID_FIELDS`.
- [ ] 自动保存 prefab.
- [ ] 返回 `modifiedFields` 和修改后值.

#### 4.4.2 `prefab.setRectTransform`

**用途**: 批量修改 RectTransform 的 UI 布局字段.

**可写字段**:

- `anchorMin`(x,y)
- `anchorMax`(x,y)
- `pivot`(x,y)
- `anchoredPosition`(x,y)
- `sizeDelta`(x,y)
- `offsetMin`(x,y)
- `offsetMax`(x,y)

**要求**:

- [ ] 目标对象必须有 `RectTransform`,否则返回 `COMPONENT_NOT_FOUND`.
- [ ] 修改前后必须满足 `anchorMin.x <= anchorMax.x` 与 `anchorMin.y <= anchorMax.y`.
- [ ] 未提供任何可写字段时,返回 `INVALID_FIELDS`.
- [ ] 自动保存 prefab.
- [ ] 返回 `modifiedFields` 和修改后值.

### 4.5 批量事务编辑

#### 4.5.1 `prefab.batchEdit`

**用途**: 在同一个 prefab 编辑上下文中串行执行多个子操作,最后统一保存.

**允许包含的子操作**:

- `prefab.setGameObjectProperties`
- `prefab.renameGameObject`
- `prefab.createGameObject`
- `prefab.deleteGameObject`
- `prefab.moveOrCopyGameObject`
- `prefab.setSiblingIndex`
- `prefab.addComponent`
- `prefab.setComponentProperties`
- `prefab.deleteComponent`
- `prefab.setTransform`
- `prefab.setRectTransform`

**要求**:

- [ ] `operations[].params` 中禁止再次出现 `prefabPath`.
- [ ] 子操作按输入顺序严格串行执行.
- [ ] `mode=stopOnError` 时,后续未执行子操作返回 `SKIPPED`.
- [ ] `mode=continueOnError` 时,可以继续执行后续子操作,但只要任一子操作失败,整体仍不得保存.
- [ ] 子操作在 batch 内不得隐式保存.
- [ ] 结果中必须返回逐步结果与整体 `saved`.

## 5. 举例覆盖需求和边缘情况

### 5.1 例 1: 先查看再改名

1. 先使用 `prefab.queryHierarchy` 找到目标对象路径和 `siblingIndex`.
2. 再调用 `prefab.renameGameObject` 改名.
3. 再次查询层级,确认 `newPath` 正确.

### 5.2 例 2: 创建新按钮并补齐组件

1. 使用 `prefab.createGameObject` 在 `DialogMain/Panel_Content` 下创建 `UIButton_New`.
2. 使用 `prefab.addComponent` 添加 `UnityEngine.UI.Button`.
3. 使用 `prefab.setRectTransform` 设置布局.
4. 使用 `prefab.setComponentProperties` 设置按钮组件属性.

### 5.3 例 3: 同名对象的精确定位

假设 `DialogMain/Panel_Content` 下有两个同名 `UIButton_Confirm`:

- 第一个对象使用 `objectPath="DialogMain/Panel_Content/UIButton_Confirm", siblingIndex=0`
- 第二个对象使用 `objectPath="DialogMain/Panel_Content/UIButton_Confirm", siblingIndex=1`

调用方不得只靠路径字符串猜测目标对象.

### 5.4 例 4: 同类型多组件的精确定位

假设同一个对象上挂了两个同类型组件:

- 先用 `prefab.queryComponents` 查询组件列表.
- 再通过 `componentType + componentIndex` 精确指定目标组件.
- 例如,删除第二个同类型组件时,使用 `componentIndex=1`.

### 5.5 例 5: 只调整排序,不移动父节点

当只是想修改同一父节点下的显示顺序时:

- 应使用 `prefab.setSiblingIndex`
- 不应把纯排序操作误写成 `prefab.moveOrCopyGameObject`

### 5.6 例 6: RectTransform 校验失败

如果输入导致 `anchorMin.x > anchorMax.x` 或 `anchorMin.y > anchorMax.y`:

- 命令必须失败.
- 不应出现部分字段已写入的情况.
- 应返回非专业人士可读的错误提示.

### 5.7 例 7: batchEdit 严格事务

当一次 `prefab.batchEdit` 中包含 3 个子操作:

1. 创建节点成功.
2. 设置布局失败.
3. 改名未执行或继续执行,由 `mode` 决定.

最终要求:

- 整体 `saved=false`.
- prefab asset 文件必须保持未修改状态.
- 调用方可通过逐步结果知道哪一步失败.

### 5.8 例 8: 不允许编辑根节点

如果调用方把 `objectPath` 直接设置为 `rootPath` 去执行:

- `prefab.renameGameObject`
- `prefab.setSiblingIndex`
- `prefab.setTransform`
- `prefab.setRectTransform`

则命令都必须失败,并明确提示不允许编辑 prefab 根节点.

### 5.9 例 9: 查看 scene 中实例的 override 明细

假设 `Main/Canvas/ShopButton` 来源于 `Button.prefab`,但在 scene 中改了文字颜色,并新增了一个脚本组件:

- 调用 `prefabBridge.getOverrides` 后,必须能区分 `modifiedProperties` 与 `addedComponents`.
- 每条改动都必须返回稳定的 `overrideId`.
- 返回结果应让非专业人士也能看懂“颜色被改了”“新增了组件”.

### 5.10 例 10: 只把部分 override 写回 prefab

假设一个实例同时改了按钮文本,颜色,位置,现在只想把文本和颜色写回 prefab:

- 先通过 `prefabBridge.getOverrides` 拿到对应 `overrideId`.
- 再调用 `prefabBridge.applyOverrides(applyMode=selected, overrideIds=[...])`.
- 返回中必须明确 `appliedOverrideIds[]`.
- 未被选中的位置改动仍保留在 scene instance 上.

### 5.11 例 11: variant 实例默认只写回 variant

假设 `ShopButton_Gold` 来源于 `Button_Gold.variant.prefab`,其 base prefab 为 `ButtonBase.prefab`:

- 调用 `prefabBridge.applyOverrides` 时,默认只允许写回 `Button_Gold.variant.prefab`.
- 不允许在当前版本中自动越级写回 `ButtonBase.prefab`.
- 返回中必须明确 `applyTargetPrefabPath`.

### 5.12 例 12: 通过框架级 commands[] 批量 apply

假设同一批次中:

1. 第 1 个命令对有效 prefab instance 执行 `prefabBridge.applyOverrides`.
2. 第 2 个命令对非 prefab instance 执行相同命令.

则要求:

- 第 1 个命令成功并保存对应 prefab asset.
- 第 2 个命令失败,且明确说明目标不是 prefab instance.
- 返回中必须按框架级 `results[]` 逐项列出状态,不允许给出模糊总结.

## 6. 与其他需求文档的关系

- `03_预制体查看.md`: 提供所有编辑操作所依赖的查看与定位能力.
- `01_整体与框架需求.md`: 提供批次协议,错误处理与整体架构约束.

## 7. 验收口径

- [ ] 以上 12 个编辑命令均有清晰且一致的输入输出语义.
- [ ] 调用方可仅通过 `unity-prefab-view` + `unity-prefab-edit` 完成常见 prefab 编辑闭环.
- [ ] 同名对象可通过 `objectPath + siblingIndex` 稳定定位.
- [ ] 同类型多组件可通过 `componentType + componentIndex` 稳定定位.
- [ ] batchEdit 满足严格事务语义,失败时不产生部分落盘.
- [ ] 能稳定查看 scene 中 prefab instance 的 overrides,且每条改动都包含可回传的 `overrideId`.
- [ ] 能将全部或部分 overrides apply 回 prefab asset.
